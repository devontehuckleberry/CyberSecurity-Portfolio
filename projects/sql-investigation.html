SQL Filters & Joins — Security Investigation Queries

SQLMariaDBLinuxLog QueryingAND / OR / NOT / LIKE
Scenario. A work sample from the Google Cybersecurity Professional Certificate. The security team needs data to investigate potential issues and to push a system update. The task: pull specific records about employees, their machines, and their departments from a MariaDB database — translating each investigation need into precise filter logic.

01After-hours failed login attempts

QUERY 01Failed logins after 18:00AND

Both conditions had to be true at the same time — the attempt happened after hours and it failed — so AND was the right operator. It returns only the rows matching every specified condition, isolating exactly the suspicious activity.

SELECT * FROM log_in_attempts
WHERE login_time > '18:00:00' AND success = 0;

02Login attempts on specific dates

QUERY 02Activity on two investigation datesOR

The suspicious event could have occurred on either of two dates under investigation, so OR was necessary — it returns rows matching either condition rather than requiring both.

SELECT * FROM log_in_attempts
WHERE login_date = '2022-05-08' OR login_date = '2022-05-09';

03Login attempts outside of Mexico

QUERY 03Exclude one country by patternNOT + LIKE

The pattern 'MEX%' catches both 'MEX' and 'MEXICO' as they appear in the dataset, and NOT reverses the match to return every other country.

SELECT * FROM log_in_attempts
WHERE NOT country LIKE 'MEX%';

04Employees in Marketing (East building)

QUERY 04Department + location subsetAND + LIKE

Combining both conditions targeted only the specific subset of employees whose machines needed the update — Marketing staff located in the East building.

SELECT * FROM employees
WHERE department = 'Marketing' AND office LIKE 'East%';

05Employees in Finance or Sales

QUERY 05Two departments, one updateOR

Finance and Sales needed a different security update from the rest of the company, so OR pulled both groups in a single query instead of running two separate commands.

SELECT * FROM employees
WHERE department = 'Finance' OR department = 'Sales';

06All employees not in IT

QUERY 06Everyone outside one departmentNOT

The update only affected non-IT staff, so NOT department = 'Information Technology' returned the full list of employees who still needed it applied.

SELECT * FROM employees
WHERE NOT department = 'Information Technology';

Summary

This project used SQL filters in MariaDB to retrieve targeted data from login and employee records in support of a security investigation and a system-update effort. I applied AND, OR, and NOT — along with LIKE for pattern matching — to isolate specific records across six scenarios. Each query meant translating an investigation need into precise filter logic, which reinforced how operator choice directly shapes what data comes back.

The bigger takeaway: SQL is a practical, everyday tool for security analysts, not just database administrators. Querying for failed logins, suspicious date ranges, geographic anomalies, and department-specific records maps directly onto real SOC and incident-response work.
